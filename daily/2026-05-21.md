## 今日课程内容

### 1. 一页摘要：AI + Web3 的核心不是「发币」

今天课程最重要的观点是：

> AI + Web3 的重点不是「AI 怎么发币」，而是当 AI 从回答问题走向执行任务时，它需要一套可授权、可支付、可记录、可追责的经济基础设施。

AI 提供理解、规划、分析和执行能力；Web3 提供钱包、链上身份、支付、验证和激励机制。两者结合之后，agent 才可能从单纯的信息工具，逐渐变成能参与链上经济行为的执行者。

课堂把 AI 与 Web3 的关系拆成双向赋能：

- **Web3 赋能 AI**：去中心化算力、数据、模型和激励网络，可能成为 AI 基础设施的一部分，帮助 AI 服务摆脱单一中心化平台限制。
- **AI 赋能 Web3**：AI 可以帮助用户理解复杂链上数据、分析风险、模拟交易，并在授权范围内执行链上任务。

一句话理解：

> Web3 让 AI 拥有经济账户和可验证协作网络；AI 让 Web3 变得更可读、更可用、更自动化。

### 2. 方向一：去中心化 AI 基础设施

代表例子：**Bittensor**。

课堂中用 Bittensor 的链上浏览器和 taostats 页面做例子，说明它和普通区块链浏览器关注点不同：

- 普通区块链浏览器更关注交易、区块和地址。
- Bittensor 更关注子网、节点、验证者、算力服务和评分激励。

我的理解是，Bittensor 可以被看作一个去中心化 AI 服务网络：

1. 每个子网对应一种 AI 能力或 agent 服务方向，例如生图、代码、模型推理等。
2. 节点提供本地部署的模型、算力或服务能力。
3. 用户通过 API 或网关提交请求。
4. 网关把请求分发给对应子网和节点。
5. 验证者持续评估节点服务质量。
6. 服务质量评分和激励分发等状态记录到链上。
7. 节点根据质量评分获得代币激励。

关键点是：真正上链的不是每一次模型推理过程，而是服务质量评分、激励分发等可验证状态。模型推理本身主要还是发生在链下。

但这个方向也有现实瓶颈：分布式节点的硬件、模型调教、上下文完整性、响应速度和稳定性参差不齐，所以短期体验通常不如中心化 AI 服务。它更像一个方向性基础设施，还不是完全成熟的替代品。

### 3. 方向二：AI Agent + 钱包

代表例子：**Coinbase AgentKit**。

这个方向回答的问题是：

> AI 怎么从「建议者」变成「执行者」？

没有钱包的 agent，通常只能分析、推荐、生成内容；接入钱包之后，agent 才能在授权范围内完成支付、转账、swap、合约调用、部署等链上动作。

课堂中提到，AgentKit 类似给 agent 接入一组链上 action，让它可以在钱包和授权约束下执行任务。

典型流程可以理解为：

1. 用户给出目标，例如购买信息、调用 API、完成链上付款或执行交易。
2. LLM 把目标拆成任务。
3. AgentKit 选择合适的链上 action。
4. Wallet Provider 负责签名并广播交易。
5. 链上结果返回给 agent。

核心看点是：如何把 LLM 的工具调用连接到真实链上动作。

但这也是风险最大的地方。钱包不只是一个登录工具，它代表资产账户、链上身份、支付权限和执行权。所以安全形态不能是「完全自治」，而应该是：

> 在人类授权范围内自动执行。

必要的 guardrails 包括：预算、白名单、单笔上限、会话上限、KYT、交易模拟、操作日志、撤销机制和关键动作人工确认。

### 4. 方向三：AI 链上分析工具

代表例子：**Arkham**。

链上数据虽然公开，但不等于容易理解。区块浏览器里的交易、地址、合约调用、资金流，对初学者甚至很多开发者来说都很难快速判断含义。

AI 链上分析工具的价值，是把链上原始数据翻译成人能理解的行为、身份线索和风险信号。

课堂中提到，Arkham 会抓取多条链和区块浏览器数据，分析地址之间的转账、交易所流入流出、钱包行为和实体归属，从而把匿名地址归类为某个个人、组织或机构相关地址。

这个方向的用途包括：

- 观察交易所或 DeFi 协议中的资金流入流出。
- 识别地址行为和潜在实体归属。
- 为量化策略、市场监控和项目研究提供数据输入。
- 把复杂链上行为转成可读解释。

需要注意的是：链上标签和行为分析只是概率判断，不是法律事实。它提升了链上数据可读性，但也天然和隐私叙事存在张力。

### 5. 方向四：智能钱包与安全助手

这是讲师认为很有机会的方向。

链上资产一旦误转、误签，或者和恶意合约交互，损失通常很难追回。现在一些钱包会做地址黑名单或基础风险提示，但还没有出现足够亮眼的 AI 钱包安全产品。

更理想的产品形态是：

- 在签名前读取目标地址、call data、授权范围和合约交互路径。
- 在 fork 模拟环境中预演交易结果。
- 判断目标合约是否恶意，或是否可能导致资金被转走、授权被滥用。
- 用自然语言告诉用户这笔交易真正会发生什么。
- 在必要时阻止高风险签名，或要求二次确认。

我的理解是，如果把「AI 交易理解」和「钱包执行能力」结合起来，就可能同时降低 Web3 使用门槛和资金风险。这和我今天自学的 Revoke.cash 也有关：用户不仅需要知道过去授权了什么，也需要在签名前知道未来会发生什么。

### 6. 方向五：交易所与 DeFi 智能风控

这一方向更多面向机构和交易所。AI 可以用于交易所监管、异常交易识别、DeFi 参数监控、攻击预警、系统风险识别等场景。

它和前面几个方向的差异是：

- 面向对象从普通用户变成机构、交易所、协议方。
- 目标从「帮助用户看懂」变成「持续发现异常」。
- 更依赖规则、模型、链上监控、KYT 和交易行为数据。

课堂没有展开太深，但它构成了 AI + Web3 的机构风控层。

## 项目矩阵：按能力场景筛选

今天课程给了一个很有用的判断方法：看到一个 AI + Web3 项目时，不要先看概念包装，而是先判断它属于哪一层，以及是否真的解决了这一层的核心问题。

我整理成五层：

1. **资源层**  
   解决「AI 服务从哪里来」的问题。代表方向包括 Bittensor、去中心化算力 / 模型网络。判断重点是节点质量、验证机制、网关分发、激励是否闭环。

2. **行动层**  
   解决「agent 如何执行链上动作」的问题。代表方向包括 Coinbase AgentKit、MoonPay Agents、Privy Agentic Wallets。判断重点是钱包接入、授权边界、支付与交易执行能力。

3. **认知层**  
   解决「如何看懂链上数据」的问题。代表方向包括 Arkham、链上分析平台。判断重点是地址标签、资金流解释和行为分析是否可靠。

4. **用户保护层**  
   解决「如何避免误签和被骗」的问题。代表方向包括 Blockaid 类安全工具、钱包插件、安全助手。判断重点是交易模拟、风险解释、恶意合约识别和用户体验。

5. **机构风控层**  
   解决「如何持续发现异常」的问题。代表方向包括交易所 / DeFi 风控、KYT、监控系统。判断重点是实时性、误报率、规则与模型结合、响应机制。

## 风险边界

今天课程反复强调：AI + Web3 最大的问题不是能不能执行，而是执行权如何被限制。

我记录了几个常见误区和更稳妥的判断：

- **误区：模型聪明，所以可以直接给执行权。**  
  更稳妥的判断：模型输出不是授权。执行权必须经过钱包、额度、白名单和人工确认约束。

- **误区：链上标签等于事实。**  
  更稳妥的判断：标签只是概率推断，必须保留不确定性和申诉空间。

- **误区：去中心化就一定更好。**  
  更稳妥的判断：去中心化资源也要验证质量；性能、稳定性和上下文完整性都是现实瓶颈。

- **误区：钱包体验越自动越好。**  
  更稳妥的判断：钱包自动化不能牺牲可解释性和签名前确认。

- **误区：AI 能替代风控规则。**  
  更稳妥的判断：AI 适合补充分析，核心资金风险仍需要规则、监控、日志和人工兜底。

## 自学补充：Revoke.cash 与钱包授权安全

今天自学了 Revoke.cash。我的理解是：

> Revoke.cash 是一个钱包权限体检工具，用来检查并取消过去给各种 Web3 网站或合约留下的 Token / NFT 授权。

在 Web3 里，用过很多网站之后，经常会遇到 Approve / 授权。这个授权并不是「我现在付钱」，而是：

> 我允许这个合约以后可以动用我钱包里的某种币或某些 NFT。

问题是，很多授权用完之后不会自动消失。可能出现这种情况：我半年前用过某个网站，早就不用了，但它的合约现在仍然有权限动用我钱包里的 USDT、USDC 或 NFT。

Revoke.cash 可以帮我检查：

- 我的钱包曾经授权过哪些网站或合约。
- 哪些授权还有效。
- 哪些授权额度很大，甚至是无限授权。
- 哪些授权已经不用了，可以取消。
- 有没有授权给被攻击过的危险合约。

一个生活类比是：以前给很多 App 开过「自动扣款权限」，有些 App 已经不用了，但理论上还可以继续扣款。Revoke.cash 就像一个「自动扣款权限管理器」，帮我看哪些 App 还有扣款权，然后把不用的取消。

今天最重要的安全意识是：

> 断开钱包连接不等于取消授权。

在 MetaMask 里点 Disconnect，只是不让那个网站继续看到我的钱包连接状态；但链上授权还在。真正要取消权限，需要用 Revoke.cash 这类工具发起一笔撤销授权交易。撤销授权通常也需要支付 Gas。

Revoke.cash 不是钱包，不是交易所，也不是追回被盗资产的地方。它主要是一个钱包权限体检和授权撤销工具。

## 今日关键概念

1. **AI + Web3**：不是简单叠加热词，而是让 AI 在可授权、可支付、可验证、可追责的边界内完成链上任务。
2. **去中心化 AI 基础设施**：用分布式节点、验证者和激励机制提供 AI 服务能力。
3. **Subnet**：Bittensor 中按能力划分的子网，每个子网可以对应一种 agent 或模型服务方向。
4. **Validator**：对节点服务质量进行评分和验证的角色，评分结果影响激励分发。
5. **Gateway**：把用户 API 请求分发到不同子网或节点的入口。
6. **AgentKit**：把钱包和链上 action 打包给 agent 使用的工具层。
7. **Wallet Provider**：负责签名和广播交易的钱包能力提供方。
8. **Call Data**：合约调用携带的数据，决定交易实际会执行什么。
9. **Fork 模拟**：在模拟链环境里预演交易结果，用于签名前风险判断。
10. **KYT**：Know Your Transaction，围绕交易和地址风险的合规 / 风控分析。
11. **Revoke.cash**：检查并撤销 Token / NFT 授权的钱包权限体检工具。
12. **Disconnect**：断开网站和钱包的连接，但不等于撤销链上授权。
13. **Unlimited Approval**：无限授权，可能让某个合约长期拥有动用某种 Token 的权限。

## 卡点 / 问题

- Bittensor 的验证者具体如何判断一个节点的 AI 服务质量？评分机制是否容易被刷分或攻击？
- AgentKit 这类工具在真实执行链上交易时，如何设计最小权限原则？
- Call Data 对新手来说很难读懂，AI 钱包安全助手如何把它准确翻译成自然语言？
- Fork 模拟能发现哪些风险？有哪些风险即使模拟也发现不了？
- 使用 Revoke.cash 撤销授权时，如何判断哪些授权可以安全撤销，哪些授权仍然需要保留？

## 课后可落地的学习路径

1. 先选一个具体项目方向，不只停留在概念层。例如：钱包安全助手、链上资金流分析、agent 自动支付、DeFi 风控监控。
