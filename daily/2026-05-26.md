我现在可以把 Nonce 理解成：同一个钱包地址发交易时的交易序号。它像钱包发交易的排队号码，用来保证交易按顺序处理，也防止重复交易。

我也理解了，Gas 是网络执行费，不是成功费。即使交易最后失败，只要网络已经尝试执行、验证和运行合约逻辑，就可能消耗计算资源，所以通常也会扣 Gas。

另外，我今天进一步理解了 Value = 0 不代表交易没有影响。Value 只表示这笔交易直接带了多少原生币；很多真正重要的影响，可能发生在 Token Transfers、NFT Transfers、授权记录、日志和输入数据里。

一句话总结：

> Nonce 看交易顺序，Gas 看网络执行成本，Value 只看主币，合约操作的真正影响要结合 Method、Token Transfers、NFT Transfers、Logs 和 Input Data 来判断。

## 一、Nonce 是什么？

Nonce 可以理解成：这个钱包地址发出的交易序号。

同一个地址发交易，必须按顺序来：

```text
第 1 笔交易：nonce = 0
第 2 笔交易：nonce = 1
第 3 笔交易：nonce = 2
```

Nonce 的作用是防止重复交易，也保证交易顺序。

人话理解：

> Nonce 就像你钱包发交易的排队号码。前一笔没处理完，后一笔可能会卡住。

所以有时候看到交易一直 Pending，可能是因为前面某个 nonce 的交易还没完成。后面的交易不能插队，只能等前面的交易先处理完。

## 二、为什么交易失败也可能扣 Gas？

因为链上已经帮你“尝试执行”了。

比如调用一个合约时，节点需要：

- 检查签名。
- 读取链上数据。
- 运行合约逻辑。
- 判断条件是否满足。
- 记录执行结果。

即使最后失败了，比如余额不够、条件不满足、滑点太高，或者合约主动 revert，网络也已经消耗了计算资源。

所以可以这样理解：

```text
交易没成功 ≠ 网络没干活
网络干了活 = 可能要收 Gas
```

但这里也要区分不同情况：

- 交易还在 Pending，没进区块：通常还没真正扣 Gas。
- 交易进区块后执行失败：通常会扣 Gas。
- 交易被替换或取消：最终只会按真正上链的那笔算。

人话理解：

> Gas 是执行费，不是成功费。

## 三、为什么很多合约操作 Value = 0，但仍然很重要？

因为 Value 只表示：这笔交易直接带了多少原生币。

在以太坊上，Value 看的是 ETH。但很多重要操作并不是直接转 ETH，而是在调用合约。

例如：

- Approve：授权合约未来使用你的 Token，Value 通常是 0。
- Swap：可能是用 USDC 换 ETH，Value 可能是 0，但 Token 已经发生变化。
- Mint：有些免费 Mint，Value 是 0，但你拿到了 NFT。
- 取消授权：Value 是 0，但权限被改掉了。

所以不能只看 Value = 0 就以为“这笔交易没事”。

更重要的是看：

```text
Method / 调用方法
Token Transfers / Token 转移
NFT Transfers / NFT 转移
Logs / 事件日志
Input Data / 输入数据
```

一句话总结：

> Value 只看主币，合约操作真正的影响常常藏在 Token 转移、授权和日志里。

## 四、Approve 授权交易怎么看？

Approve 的核心是：

> 我授权某个合约未来可以使用我的 Token 或 NFT。

在区块浏览器里重点看：

```text
Status：是否成功
Method：是不是 Approve / SetApprovalForAll
To：通常是 Token 合约或 NFT 合约
Logs：有没有 Approval 事件
Spender：被授权的是谁
Amount：授权额度是多少
```

Approve 通常不会马上转走 Token，所以 Token Transfers 可能没有变化。

但它很重要，因为它是在给某个合约“未来花你 Token 的权限”。

特别需要小心：

```text
Unlimited / 无限授权
SetApprovalForAll / 授权整个 NFT 系列
陌生 Spender 地址
```

一句话总结：

> Approve 看授权对象和授权额度，不是只看有没有 Token 转出。

## 五、Swap 换币交易怎么看？

Swap 的核心是：

> 我用一种 Token 换另一种 Token。

在区块浏览器里重点看：

```text
Status：是否成功
To：调用的是哪个 Router / 聚合器 / 合约
Method：Swap / Multicall / Execute 等
Token Transfers：你付出了什么 Token，收到了什么 Token
Internal Transactions：有没有 ETH / WETH 相关流动
Transaction Fee：手续费花了多少
```

Swap 不要只看 Value。

如果用 USDC 换 ETH，Value 可能是 0，但 Token Transfers 会显示 USDC 转出，以及 ETH / WETH 或其他 Token 转入。

人话理解：

> 看 Swap，重点看 Token Transfers：出去什么，回来什么。

## 六、Mint 铸造交易怎么看？

Mint 的核心是：

> 我调用合约生成 / 领取一个新的 Token 或 NFT。

在区块浏览器里重点看：

```text
Status：是否成功
To：NFT 合约或 Mint 合约
Method：Mint / SafeMint / Claim 等
Value：有没有支付 mint 价格
NFT Transfers：有没有 NFT 转到你的地址
Token ID：你拿到的是哪个 NFT 编号
Logs：有没有 Transfer 事件
```

NFT Mint 最关键的是看：

```text
From: 0x000...000
To: 你的地址
Token ID: 某个编号
```

这通常表示这个 NFT 是新铸造出来并发给你的。

一句话总结：

> Mint 看 NFT Transfers 和 Token ID，确认是不是有新的 NFT 被铸造到你的地址。

## 七、今天上课学到：MPC、Pact 和 Recipe

今天上课还学到了 Agentic Wallet 相关内容：MPC 只是底层资金安全，Agent 真正代表用户执行链上动作时，还需要一套明确授权机制。

### 1. MPC 只是底层资金安全，Agent 还需要明确授权

MPC 钱包可以保证没有单一方能够私自动用用户资金，但这并不等于 Agent 可以随便代表用户做链上操作。

如果 Agent 要真正执行链上动作，还需要一份明确授权，告诉 Agent：

- 能做什么。
- 不能做什么。
- 可以在哪些链、Token、合约、金额范围内操作。
- 哪些环节必须停下来让用户审阅或批准。

一句话理解：

> MPC 解决“资金不能被单点私自动用”的问题，Pact 解决“Agent 到底被允许做什么”的问题。

### 2. Pact：Agent 的授权协议

Pact 是位于钱包之上的基础执行约束层，用来限制 Agent 的链上操作。

一个 Pact 主要包含四个要素：

#### Intent：用户意图

Intent 是用户希望 Agent 完成的目标或任务。

例如：

- 在 ETH 低于某个价格时买入。
- 在 ETH 高于某个价格时卖出。
- 在指定协议里执行某个 DeFi 操作。

#### Execution Plan：执行计划

Execution Plan 是 Agent 把 Intent 转成具体执行步骤。

它需要说明：

- 调用哪个合约。
- 交易数量是多少。
- Token Pair 是什么。
- 执行路径是什么。
- 可能涉及哪些链上操作。

这部分要求透明且可审计，不能只是让 Agent 模糊地说“我会帮你处理”。

#### Policy：风控约束

Policy 是 Pact 最核心的部分。

它可以限制：

- 预算。
- 审批白名单。
- 链。
- Token。
- 合约。
- ABI 参数级别的操作范围。

Cobo 的策略引擎会在每次执行时重复校验，避免 Agent 的链上行为越界。

#### Completion Condition：完成 / 失效条件

Completion Condition 用来避免 Zombie Permission，也就是长期存在、忘记撤销的权限。

Pact 不应该永久有效，可以按照这些条件自动失效或被撤销：

- 时间。
- 金额。
- 次数。
- 任务完成状态。

一句话理解：

> Pact 不只是一次授权，而是一份有边界、有风控、有失效条件的 Agent 执行协议。

### 3. Pact 的执行流程

Pact 的大致执行流程是：

1. 用户在 dApp 或 Agent 中表达执行意图。
2. Agent 与 Cobo Agentic Wallet 接口通信。
3. 意图被转成 Execution Plan，并同步组织 Policy 和 Completion Condition。
4. 这些信息被封装成 Pact，推送到移动端 App。
5. App 用自然语言展示 Pact 内容，用户审阅、修改、批准或拒绝。
6. 用户批准后，Agent 只能在 Pact 授权范围内发起链上操作。

这个流程让我理解到，Agentic Wallet 不是让 Agent 直接拿走钱包权限，而是让用户先看懂授权边界，再决定是否允许执行。

### 4. Recipe：让 Agent 把事情做对

如果说 Pact 定义边界，那么 Recipe 提供能力。

分享中把 Recipe 定义为“知识胶囊”或预置支持库，用来降低 Agent 执行复杂链上操作时的错误率。

Recipe 会打包链上任务所需的：

- 合约地址。
- ABI 参数。
- 安全风控条件。
- 经过验证的执行路径。

这样可以帮助 Agent 按固定步骤完成任务，减少大模型幻觉和随意发挥。

典型场景包括：

- Uniswap。
- Bridge。
- Hyperliquid。
- DeFi Yield。

Agent 可以调用对应 Recipe，理解应该如何更准确地执行链上动作。

一句话理解：

> Pact 负责限制 Agent 不越界，Recipe 负责帮助 Agent 少犯错、按正确路径执行。

### 5. 产品设计要点

今天还学到几个产品设计点：

- 一个用户可以创建多个钱包，并把不同钱包委托给不同 Agent。
- 不同 Agent 可以管理不同资金池，例如 Trading Agent、DeFi Yield Agent 等。
- 每个 Agent 的 Pact 和 Policy 边界可以不同，从而用更细颗粒度管理资金安全。
- 在 Pact 审阅过程中，产品可以加入 AI 风险辅助，基于用户意图和执行计划提示高 / 中 / 低风险。
- AI 风险辅助可以帮助用户理解复杂合约字段和潜在风险。

这种设计让我感觉，Agentic Wallet 的核心不是“完全自动化”，而是“在用户授权边界内自动化”。

### 6. Demo 中体现的能力

Demo 中体现了这些能力：

- Agent 发起操作后，用户会收到 Pact 审批请求。
- 用户在移动端完成审阅和授权后，Agent 才具备执行权限。
- Uniswap / Bridge、Hyperliquid 等场景都可以通过 Pact 限制执行范围。
- 对复杂链上字段，系统会做风险解释和提示。
- 用户可以识别异常合约地址、异常参数或高风险行为。

今天这部分学习让我意识到：AI Agent 和钱包结合时，最重要的问题不是“Agent 能不能执行”，而是“Agent 在什么边界内执行、谁来审阅、什么时候权限失效”。

## 打卡草稿

今天我继续学习链上交易的细节，重点理解了 Nonce、失败交易为什么也可能扣 Gas、为什么很多合约操作 Value = 0 但仍然很重要，以及 Approve / Swap / Mint 在区块浏览器里分别应该看哪里。

我今天最大的收获是理解了 Nonce。Nonce 可以理解成同一个钱包地址发交易的顺序编号，就像钱包发交易的排队号码。第 1 笔交易是 nonce = 0，第 2 笔是 nonce = 1，第 3 笔是 nonce = 2。同一个地址的交易必须按顺序处理，所以如果前一笔交易一直 Pending，后面的交易可能也会被卡住。

我也理解了 Gas 是执行费，不是成功费。交易失败不代表网络没有干活。比如调用合约时，节点已经检查签名、读取数据、运行合约逻辑，即使最后因为余额不够、条件不满足、滑点太高或合约 revert 而失败，网络也已经消耗了计算资源，所以通常仍然会扣 Gas。

今天还学到，Value = 0 不代表这笔交易没有影响。Value 只表示这笔交易直接带了多少原生币，而很多重要的合约操作，比如 Approve、Swap、Mint、取消授权，真正的影响可能发生在 Token Transfers、NFT Transfers、Logs 或 Input Data 里。

最后我整理了三类常见交易的阅读方法：Approve 要看授权对象和额度，特别注意无限授权和陌生 Spender；Swap 要看 Token Transfers，判断付出了什么、收到了什么；Mint 要看 NFT Transfers 和 Token ID，确认是不是有新的 NFT 被铸造到自己的地址。

今天上课还学到了 Agentic Wallet 里的 MPC、Pact 和 Recipe。我的理解是：MPC 主要解决底层资金安全，保证没有单一方可以私自动用用户资金；但 Agent 真正代表用户执行链上操作时，还需要 Pact 来明确授权边界，比如 Intent、Execution Plan、Policy 和 Completion Condition。Pact 可以限制 Agent 能做什么、不能做什么、在哪些链和合约里操作、预算是多少、什么时候权限失效。

另外，Recipe 像是给 Agent 使用的“知识胶囊”或预置支持库，会打包合约地址、ABI 参数、安全风控条件和经过验证的执行路径，帮助 Agent 在 Uniswap、Bridge、Hyperliquid、DeFi Yield 等复杂场景里减少幻觉和随意发挥。

我今天对 Agentic Wallet 的核心理解是：它不是让 Agent 拿到无限权限后完全自动化，而是让 Agent 在用户看得懂、可审阅、可撤销、有边界的授权范围内自动化。
