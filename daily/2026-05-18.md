## 学习记录

### 核心观点

这节课的核心是：AI 会提升编码和学习效率，但不会降低系统复杂度；Web3 不是脱离 Web2 的新世界，而是在支付、钱包、链上状态、签名和信任模型上提出了更高的安全要求。开发者真正要补的是基础知识、架构判断和 Debug 能力，做到“驾驭 AI，而不是被 AI 驱动”。

### AI 时代更需要基础知识

人的职责是设计方案、评估合理性、验收结果、分析问题；AI 的职责是辅助细化方案、生成 plan、执行编码和帮助自我验证。AI 能放大执行力，但如果人看不懂方案、无法判断实现质量，就会被 AI 带着走。

### Web3 和 Web2 不是割裂关系

很多 Web3 项目仍然大量依赖 Web2 能力。比如 CEX 有很重的 Web2 账户、风控、撮合和后台体系；DeFi 也需要前端、后端状态管理和良好 UX。Web3 只是把某些核心信任、资产和状态逻辑放到了链上。

### Web3 安全必须“源于设计”

Web2 项目常常是“先跑起来，再补安全”；但 Web3 涉及资金交互，一旦设计错误，可能上线即资产归零。审计只是锦上添花，不能代表安全本身。真正的安全要贯穿需求、架构、开发、上线、运维和风控全生命周期。

### 传统 Web2 支付模型

用户在电商平台下单，平台创建支付订单，支付服务向银行机构获取转账信息，展示给用户；用户转账后，银行回调支付服务，支付服务再回调电商平台，订单进入履约。

资金流是：

```text
用户银行卡 -> 支付服务账户 -> 商户账户
```

这里的安全依赖 API key、secret key、支付密码、MFA、风控，以及对银行和支付服务的信任。

### Web3 支付模型

把传统模型里的 Bank Institution 替换成 Web3 Blockchain。用户向 deposit account 转 USDC，链上交易被记录；链上监听服务 pull/listen 到交易后，确认到账，再通知支付服务。

关键区别是：区块链不会主动 callback 外部系统，所以必须有链上监听服务。

```text
用户
  ↓
电商平台下单
  ↓
支付服务
  ↓
生成 deposit account
  ↓
用户钱包转 USDC
  ↓
Web3 Blockchain
  ↓
链上监听服务
  ↓
确认到账后通知支付服务 / 业务系统
```

### 区块链解决的是信任问题

Web2 依赖中心化机构的账本和回调；Web3 依赖不可篡改的数据、公开透明且确定性执行的链上逻辑，以及 PoW、PoS、DPoS、PoH 等共识机制。不同链的实现不同，但目标都是减少对单一权威的依赖。

### 钱包的本质

钱包不是“装币的容器”，而是管理私钥并产出签名的工具。私钥是一串随机数，配合 ECDSA、EdDSA 等算法生成 signature。交易签名表达“我要移动资产”的意图；消息签名可用于身份认证。链认私钥和签名，不认现实中的人。

### 钱包安全重点

私钥泄露等于资产控制权丢失；私钥忘记也无法像银行卡密码一样重置。常见钱包形态包括自托管钱包、全托管钱包、混合托管钱包、MPC 钱包、合约钱包、多签钱包、硬件钱包、隐私钱包、分层钱包。大额资金通常要用多签、权限拆分、冷热钱包和更强的签名环境。

### 签名欺骗与交易模拟

用户看到的“签名内容”可能和真实交易效果不一致，所以需要签名可视化，例如 EIP-712，也需要交易 simulation：在签名前模拟执行，查看哪些账户、余额、授权会变化。服务侧还会结合多签、TEE 可信执行环境、MPC 等方式降低私钥明文暴露和单点风险。

### 交易生命周期

一笔链上交易通常经历：

```text
构造交易 -> 私钥签名 -> 广播到链 -> 节点共识 -> 状态更新
```

关键参数包括 gas、nonce、call data / instructions。监听到账时还要考虑链分叉和 reorg，所以不能看到一笔交易就立刻上账，需要等待一定 confirmation / finality，再做 KYT、地址风险、合规检查，最后 callback 支付服务。

## 关键概念

1. AI 是放大器，不是系统复杂度的消除器。
2. Web3 与 Web2 不是割裂关系，很多系统仍然依赖 Web2 工程能力。
3. Web3 支付需要链上监听服务，因为区块链不会主动 callback 外部系统。
4. 钱包的本质是私钥管理和签名工具，不是资产容器。
5. Web3 安全要从需求和架构设计开始，而不是上线前只靠审计。
6. 交易确认要考虑 confirmation、finality、分叉和 reorg。

今天重点记录了几个概念：

- Web3 安全必须源于设计，不能只靠上线前审计。
- 传统支付依赖银行和支付服务的回调，Web3 支付则需要链上监听服务，因为区块链不会主动 callback 外部系统。
- 钱包不是“装币的容器”，而是管理私钥并产出签名的工具。
- 私钥泄露等于资产控制权丢失，不能像银行卡密码一样重置。
- 链上交易需要经历构造、签名、广播、共识和状态更新，到账确认还要考虑 confirmation、finality、分叉和 reorg。

我当前的卡点是：还需要继续补充共识机制、EIP-712 签名可视化、交易 simulation、链上监听服务等更具体的工程实现。后续学习会优先补 Web3 基础，再结合 AI 做学习辅助、方案设计和代码验证。
