## 今日学习内容

### 1. 点击 Confirm 之后，链上发生了什么

今天课程围绕一个问题展开：

> 当用户在钱包里点击 Confirm 之后，链上到底发生了什么？

我的理解是，这不是一个简单的“发送成功”动作，而是一整条从本地钱包到去中心化网络的流程。

大致主线是：

1. 钱包在本地保存私钥，并帮助用户构造交易。
2. 用户确认后，钱包用私钥对交易进行签名。
3. 签名后的交易通过 RPC 广播到区块链网络。
4. 交易进入 mempool，等待被排序和打包。
5. builder 根据费用等因素选择交易并组织区块内容。
6. validator 验证交易、参与出块和共识。
7. 区块被确认后，交易状态逐步进入 safe / finalized 等更稳定的状态。
8. 如果交易调用了智能合约，合约代码会被执行，并改变链上状态。

这让我意识到：钱包界面里一个很简单的 Confirm，背后其实连接了密码学签名、网络传播、经济激励、区块生产和共识机制。

### 2. 钱包、私钥、助记词和地址

今天也从钱包、私钥、助记词和地址讲起。我目前的理解是：

- **钱包** 不是真的“存钱的地方”，更像是管理私钥、构造交易和发起签名的工具。
- **私钥** 是控制链上资产和身份的关键，谁掌握私钥，谁就能发起有效签名。
- **助记词** 通常用于恢复钱包，本质上也是极其敏感的恢复材料。
- **地址** 是链上的公开标识，可以用来接收资产、查看交易记录和与合约交互。

这里最重要的安全意识是：私钥和助记词不能公开、不能截图乱传、不能提交到 GitHub，也不能输入到不可信网站。

### 3. 交易、签名、RPC 和 mempool

今天我把交易流程理解成：

1. 钱包先根据用户操作构造交易，例如转账、Swap、Mint NFT、授权 Token、调用合约等。
2. 用户确认后，钱包用私钥签名。签名可以证明“这笔交易确实由这个地址授权”。
3. 签名后的交易通过 RPC 节点发送到区块链网络。
4. 交易会先进入 mempool，也就是待处理交易池，等待被打包进区块。

这里我对 RPC 的理解是：它像钱包和区块链网络之间的接口。钱包不一定自己运行完整节点，所以需要通过 RPC 把交易发出去，也通过 RPC 查询链上状态。

### 4. builder、validator、区块确认和共识

交易进入 mempool 后，并不是马上最终完成。课程里还讲到 builder、validator、区块确认和共识机制。

我的理解是：

- **builder** 会根据费用、交易排序等因素，把交易组织成候选区块内容。
- **validator** 负责验证区块和参与出块。
- 交易被打包进区块后，通常会显示为 confirmed，但这并不等于立刻达到最高确定性。
- 随着时间推进，区块可能进入 **safe** / **finalized** 状态，表示它越来越难被重组或回滚。
- 共识机制让很多独立节点在没有中心化管理员的情况下，对同一份链上状态达成一致。

这部分让我开始理解 Web3 的“可信”不是来自某一个平台承诺，而是来自密码学、经济激励、节点验证和社会共识共同作用。

### 5. 智能合约和链上状态变化

今天还学习了智能合约。我的理解是：

> 智能合约是部署在链上的代码，用户通过交易触发它执行，执行结果会改变链上状态。

比如：

- Swap 会调用去中心化交易所合约，改变用户和池子的 Token 余额。
- Mint NFT 会调用 NFT 合约，生成或分配新的 NFT。
- Token Approval 会调用 Token 合约，记录某个合约或地址未来可以使用我的 Token 额度。

这让我意识到：链上交易不只是“转账”，很多时候是在调用代码、改变状态、留下公开可验证的记录。

## 自学补充

### 1. Gas 是什么

今天自学了 Gas。我的理解是：

> Gas 是以太坊里的“手续费计量单位”，用来衡量一笔链上操作需要消耗多少计算资源。

在以太坊上，只要是会改变链上状态的操作，通常都需要支付 Gas，例如：

- 转账
- Swap
- Mint NFT
- 授权 Token
- 调用智能合约

几个相关概念：

- **Gas**：这次操作需要多少“工作量”。
- **Gas Price / Gas 价格**：愿意为每一单位 Gas 支付多少钱，常用 Gwei 表示。
- **Gas Fee / 矿工费 / 手续费**：最终支付的费用。

大致可以理解为：

```text
Gas Fee ≈ Gas 用量 × 每单位 Gas 价格
```

例如：

- 普通 ETH 转账通常需要 21,000 gas。
- 调用复杂合约，比如 Swap，可能需要更多 gas。
- Gas 通常用 ETH 支付。

我今天也理解到：即使交易失败，Gas 也可能被扣掉。因为链上已经为这笔交易做了执行和验证，只是最终执行结果没有达到预期。

Gas 的作用主要有三个：

1. **防止垃圾交易**：如果链上操作完全免费，就容易被大量无意义交易攻击。
2. **限制无限循环**：合约执行需要消耗 Gas，可以避免程序无限执行拖垮网络。
3. **激励验证者处理交易**：手续费让验证者有动力维护网络运行。

### 2. Token Approval 为什么有风险

今天还自学了 Token Approval。我的理解是：

> Token Approval 是“授权”，不是直接转账，而是允许某个合约或地址在未来最多动用我某种 Token 的一定数量。

比如在 Uniswap 用 USDT 换 ETH，常见流程是：

1. 先授权 Uniswap 合约可以使用我的 USDT。
2. 再进行 Swap，Uniswap 合约从我的钱包中扣走相应数量的 USDT。

风险在于：授权一旦给出去，通常会一直存在，直到我主动撤销或把额度改成 0。

常见风险包括：

1. **无限授权风险**  
   很多网站会请求 Unlimited 授权，也就是几乎无限额度。这样以后这个合约理论上可以花掉我钱包里所有该 Token。

2. **恶意合约风险**  
   如果授权给假网站或钓鱼合约，对方可能直接转走授权范围内的 Token。

3. **合约被攻击风险**  
   即使原项目是真的，如果合约有漏洞、被攻击或被恶意升级，也可能利用之前留下的授权。

4. **旧授权遗忘风险**  
   很久以前用过的 DApp，授权可能还在。虽然我已经忘记，但它仍然可能有权限。

5. **NFT 授权更危险**  
   有些 NFT 授权是 `setApprovalForAll`，意思是允许对方操作整个系列的 NFT，而不只是某一个 NFT。

我今天对 Approval 的安全理解是：授权不是小事。以后连接钱包和点击 Confirm 前，要特别看清楚授权对象、授权额度和授权类型。

## 今日关键概念

1. **Confirm**：钱包中用户确认交易或授权的动作，背后通常会触发签名和广播流程。
2. **私钥 / 助记词**：控制钱包和链上资产的核心敏感信息，绝不能公开。
3. **签名**：用私钥证明某笔交易确实由该地址授权。
4. **RPC**：钱包和区块链网络交互的接口。
5. **mempool**：待处理交易池，交易在被打包前会先进入这里等待。
6. **builder**：组织和排序交易，构建候选区块内容的角色。
7. **validator**：验证交易和区块、参与出块与共识的角色。
8. **safe / finalized**：区块确认后更稳定的状态，代表交易被回滚的可能性越来越低。
9. **Gas**：链上计算和执行资源的手续费计量单位。
10. **Token Approval**：允许某个合约或地址未来使用我某种 Token 的授权机制。
11. **智能合约**：部署在链上的代码，被交易触发后执行并改变链上状态。
