## 今日核心总结

今天我继续理解一笔链上交易从钱包发出到被记录在区块链上的过程。

我现在可以把 RPC 理解成：钱包和区块链之间的“通信入口”。钱包本身不会保存整条区块链数据，它要查询余额、Gas、交易状态、合约信息，或者把签好的交易发送到网络，都需要通过 RPC 去和节点沟通。

一句话总结：

> RPC 不是钱包，也不是链本身；它是钱包连接节点、读取数据、发送交易的通道。

我也进一步理解了 Mempool 和 Pending。交易在钱包里点击 Confirm 后，不一定马上写进区块链账本，而是可能先进入交易等待区，也就是 Mempool。只要交易还没被打包进区块，通常就处于 Pending 状态。

一句话总结：

> Mempool 是交易等待区，Pending 是交易正在等待被打包。

## 一、RPC 是什么？

RPC 可以理解成：钱包和区块链之间的“通信入口”。

钱包本身不会直接保存整条区块链数据。它要知道余额、Gas、交易状态、合约信息，就需要通过 RPC 去问区块链网络。

RPC 全称是 Remote Procedure Call，但作为新手，我现在不需要先记英文，只要记住：

> RPC = 钱包向区块链发请求的通道。

## 二、钱包为什么需要 RPC？

因为钱包要做很多事，例如：

- 查 ETH / Token 余额。
- 查当前 Gas 价格。
- 查询交易有没有成功。
- 读取智能合约数据。
- 把签好的交易发送到网络。
- 判断当前连接的是哪条链。

比如我打开钱包看到“余额 1 ETH”，这个余额不是钱包自己算出来的，而是钱包通过 RPC 问链上节点得来的。

人话理解：

> 钱包负责签名，RPC 负责帮钱包和区块链说话。

## 三、RPC 和节点是什么关系？

节点是保存和验证区块链数据的电脑。

RPC 是访问这个节点的接口。

可以这样理解：

```text
钱包
  ↓ 通过 RPC 请求
节点
  ↓ 读取 / 验证区块链数据
区块链网络
```

节点像“数据库和验证机器”。

RPC 像“对外服务窗口”。

平时使用的钱包，通常会默认连接某些 RPC 服务商，比如 Infura、Alchemy、公共 RPC、项目方 RPC，或者用户也可以自己运行节点。

## 四、如果 RPC 挂了，我的钱会不会丢？

不会。

因为钱不在 RPC 里，也不在钱包 App 里。资产记录在区块链上。

RPC 挂了，通常只是：

- 钱包暂时查不到余额。
- 页面加载不出数据。
- 交易发不出去。
- 交易状态刷新不了。
- 显示网络错误。

但资产不会因为 RPC 挂了而消失。

如果一个 RPC 出问题，通常可以换另一个 RPC。就像一个查询窗口关了，不代表账本没了，只是暂时问不到，换一个窗口就可以继续。

不过也要注意：恶意 RPC 一般不能直接偷钱，因为它没有私钥。但它可能显示错误信息、隐藏真实状态、让交易发送失败，或者追踪地址和 IP。所以 RPC 也要尽量使用可信来源。

## 五、RPC 的生活类比

可以把区块链想成一本公共账本。

- 节点是保存这本账本的图书馆。
- RPC 是图书馆的服务窗口。
- 钱包是手里的印章工具。

要查余额，就像去窗口问：

> 我这个地址账上有多少钱？

要发交易，就像先用自己的印章签好一张申请单，然后交给窗口：

> 请把这张签好的操作提交到账本系统。

如果窗口关了，钱不会丢，只是暂时查不了、交不了申请。换一个窗口就可以继续。

## 六、Mempool / Pending 是什么？

可以把 Mempool / Pending 理解成：

> 交易已经发出去了，但还没正式写进区块链账本。

Mempool 叫“交易池”。

当我在钱包点 Confirm 之后，钱包会把签好名的交易发到网络里。但这笔交易不会立刻进区块。它通常会先进入一个等待区，这个等待区就叫 Mempool。

人话理解：

> Mempool = 交易排队区。

Pending 是“等待中”。

当一笔交易还在 Mempool 里，没被打包进区块时，它的状态通常就是 Pending。

人话理解：

> Pending = 交易正在排队，还没真正上链。

## 七、为什么交易会 Pending？

常见原因有：

- Gas 给太低，别人出的手续费更高，所以他们先被处理。
- 网络很忙，排队交易太多。
- RPC 或钱包显示延迟。
- 前一笔交易还没完成，后一笔交易要等。
- Nonce 顺序卡住了，也就是同一个地址的交易必须按顺序处理。

交易 Pending 的时候，通常还没有真正完成。它只是“申请已经提交，正在排队”。

只有交易被打包进区块并成功执行后，链上状态才真的改变。

钱包界面可能会提前显示处理中，让人感觉钱已经动了，但最终还是要看区块浏览器上的状态。

## 八、Mempool / Pending 的生活类比

可以类比成去银行办转账：

1. 填好单子并签字。
2. 把单子交到窗口。
3. 银行系统排队处理。
4. 处理成功后，账本才真正更新。

Mempool 就是窗口后面的“待处理单据筐”。

Pending 就是单子还在筐里排队。

一句话总结：

- Mempool = 交易等待区。
- Pending = 交易正在等待被打包。
- Confirmed = 交易已经进区块。
- Finalized = 交易基本不可逆。

所以看到 Pending，不要理解成“失败了”，而是：

> 交易已经发出，但还在等网络把它写进账本。

## 九、如何看懂一笔链上交易？

可以把一笔链上交易理解成一张“快递单 + 银行流水 + 执行记录”。

区块浏览器里的字段是在告诉我：

- 这笔交易是谁发的。
- 发给谁。
- 什么时候发生。
- 有没有成功。
- 花了多少钱。

### 1. Transaction Hash / 交易哈希

这是这笔交易的唯一编号。

它像快递单号、订单号。复制这个 Hash 到区块浏览器，就能查到这笔交易。

人话理解：

> Transaction Hash 用来定位这笔交易。

### 2. Status / 状态

Status 告诉我这笔交易有没有执行成功。

常见状态：

- Success：成功。
- Failed：失败。
- Pending：等待中，还没进区块。

需要注意：失败不代表没有花钱。交易失败时，因为节点已经帮忙执行和验证过，通常还是会扣 Gas。

人话理解：

> Status 看这笔操作最后成没成。

### 3. Block / 区块

Block 表示这笔交易被打包进了哪个区块。

区块像账本的一页。一笔交易只有进入某个区块，才算真正被写进链上记录。

人话理解：

> Block 看这笔交易被写进了第几页账本。

### 4. Timestamp / 时间戳

Timestamp 表示这笔交易所在区块被产生的大概时间。

人话理解：

> Timestamp 看这笔交易是什么时候被记录到链上的。

### 5. From / 发起方

From 表示这笔交易是谁发起的。

通常是钱包地址。From 地址需要用私钥签名，证明“我同意发这笔交易”。

人话理解：

> From 看谁发起了这次操作。

### 6. To / 接收方或合约

To 表示这笔交易发给了谁。

这里要注意：To 不一定是普通人的钱包地址，也可能是智能合约地址。

例如：

- 转 ETH 给朋友：To 是朋友地址。
- 在 Uniswap 换币：To 是 Uniswap 合约。
- Approve Token：To 是 Token 合约。
- Mint NFT：To 是 NFT 合约。

人话理解：

> To 看这笔操作发给了哪个地址，可能是人，也可能是合约。

### 7. Value / 转账金额

Value 表示这笔交易直接转了多少原生币。

例如：

- Ethereum 上的原生币是 ETH。
- BNB Chain 上的原生币是 BNB。
- Polygon 上可能显示 POL / MATIC，取决于显示口径。

需要注意：Value 是 0 ETH，不代表这笔交易没有意义。

很多合约操作，比如 Approve、Swap、Mint，Value 可能是 0，但它仍然可能转移 Token 或改变授权。

人话理解：

> Value 看这笔交易直接附带转了多少主币。

### 8. Transaction Fee / 手续费

Transaction Fee 是为这笔交易付出的链上手续费。

手续费给网络验证者 / 区块生产者，不是给交易对方。

人话理解：

> Transaction Fee 是为了让网络处理这笔交易而花的钱。

### 9. Gas Price / Gas Used

Gas Used 表示这笔交易实际用了多少计算量，像“这次操作消耗了多少工作量”。

Gas Price 表示每一单位 Gas 付多少钱，像“每单位工作量多少钱”。

大概关系是：

```text
Transaction Fee ≈ Gas Used × Gas Price
```

人话理解：

- Gas Used = 用了多少。
- Gas Price = 单价多少。
- Transaction Fee = 总共花了多少手续费。

## 十、看一笔交易时的阅读顺序

看一笔交易时，可以按这个顺序：

1. Status：成功还是失败。
2. From：谁发起。
3. To：发给谁 / 调了哪个合约。
4. Value：有没有直接转主币。
5. Token Transfers：有没有转 Token。
6. Transaction Fee：花了多少手续费。
7. Block / Timestamp：什么时候被记录。

一句话总结：

> Transaction Hash 是编号，Status 看结果，Block 和 Timestamp 看什么时候上链，From / To 看谁对谁做了什么，Value 看主币转账，Fee 和 Gas 看这次操作花了多少手续费。

3. 节点保存和验证区块链数据，RPC 是访问节点的接口。
4. RPC 挂了，资产不会丢，因为资产记录在链上。

6. Confirmed 表示交易已经进入区块，Finalized 表示交易基本不可逆。
7. 看交易时，先看 Status，再看 From / To / Value / Token Transfers / Fee / Block / Timestamp。
8. Value 为 0 不代表交易没意义，很多合约操作本身就可能不直接转主币。
